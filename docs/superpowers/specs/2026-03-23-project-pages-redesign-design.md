# 项目列表页与项目详情页改版设计文档

日期：2026-03-23  
项目：`Product-Manager-Portfolio`  
范围：`/projects` 列表页、`/projects/[slug]` 详情页，以及支撑这两类页面改版所需的卡片与案例展示组件  
状态：已在对话中确认方向，等待你审阅书面版

## 1. 改版目标

这次改版的目标是把项目列表页和项目详情页从“英文技术模板页”改成和首页同一套中文作品集视觉语言。

希望最终达到的效果：

- 项目列表页更像作品目录，而不是普通内容列表
- 项目详情页更像“案例说明页”，而不是技术博客模版
- 用户从首页点进项目后，视觉和阅读体验不会断层
- 原型、GitHub、外部展示等入口在项目页中更容易被看到和点击

## 2. 已确认的方向

本次方向已经在对话中确认：

- 先改 `项目列表页 + 项目卡片`
- 再改 `项目详情页`
- 保持现有内容模型不变，不重做 `src/content/projects/*.mdx` 的 schema
- 保留当前已经接入的原型 / GitHub / 外链入口，但要做更好的展示

## 3. 本次范围

### 本次包含

- 重做 `/projects` 列表页首屏与卡片墙节奏
- 重做 `ProjectCard` 的视觉层级和信息组织
- 重做 `/projects/[slug]` 页头封面区
- 把详情页章节标题全部中文化
- 统一原型按钮、GitHub 按钮、外部展示按钮样式
- 统一返回入口、状态标签、元信息表达

### 本次不包含

- 修改项目内容数据结构
- 重写项目正文内容本身
- 重做首页、联系页、职业历程页
- 新增 CMS、检索、筛选、分页等复杂系统能力

## 4. 当前问题

根据当前代码和线上表现，这组页面主要有以下问题：

- `项目列表页` 还是标准内容索引页写法，缺少作品集目录感
- `ProjectCard` 更像博客卡片，而不是作品摘要卡
- `项目详情页` 中的大部分章节标题仍然是英文
- 详情页页头较弱，无法快速说明“这是什么项目、你扮演什么角色、哪里可以直接看原型/仓库”
- 现有原型入口虽然已经接上，但分散在正文中，进入路径不够直接
- 首页已经完成了浅色编辑式风格，但项目页仍然停留在旧模板语言里

## 5. 设计原则

这次改版遵循以下原则：

- 首页与项目页必须是一套视觉系统
- 项目页优先服务“快速理解项目价值”，而不是展示模板能力
- 中文化优先，避免出现英文 case study 模板感
- 原型 / 仓库 / 外链入口前置，但不打断正文叙事
- 尽量少动结构，多做页面层与组件层重构

## 6. 项目列表页设计

### 6.1 页面定位

`/projects` 页面改造成“作品集索引页”，承担两个职责：

- 让用户快速知道你做过哪些方向的项目
- 帮用户决定下一步先点开哪个项目

### 6.2 页面结构

列表页将按下面顺序组织：

1. 顶部简短 Hero
2. 项目统计 / 方向概览
3. 项目卡片墙

### 6.3 Hero 表达

列表页顶部不再只是一个普通标题，而会变成一个简短的作品索引页头部，说明：

- 这是你的项目作品页
- 覆盖知识库、RAG、工业流程、政务、数字人、具身智能等方向
- 这些项目既有公司项目，也有个人预研与外链展示

### 6.4 卡片设计

新的项目卡会延续首页风格：

- 暖白底
- 粗描边
- 大圆角
- 轻微投影
- 小标签和元信息前置

每张卡片保留这些核心信息：

- 项目名
- 角色
- 年份 / 状态
- 一句成果摘要
- 关键方向标签

对于以下项目，卡片上增加轻量提示：

- 有原型：显示“可查看原型”
- 有 GitHub：显示“附仓库入口”
- 有外部展示：显示“附外链”

这个提示只做辅助识别，不直接把卡片做成按钮堆叠。

## 7. 项目详情页设计

### 7.1 页面定位

`/projects/[slug]` 详情页改成“中文案例页”，重点是：

- 快速告诉用户这个项目是什么
- 你在里面做了什么
- 有什么结果 / 价值
- 哪里可以点进去看原型、仓库或外链

### 7.2 页头封面区

详情页头部将重做为项目封面区，包含：

- 项目标题
- 角色
- 时间 / 状态 / 阅读时长
- 一句结果摘要
- 关键入口按钮

按钮优先级：

- 如果有原型，优先显示 `查看原型`
- 如果有 GitHub，显示 `查看仓库`
- 如果有外链，显示 `外部展示`

正文里的按钮可以保留，但页头要优先聚合出关键入口。

### 7.3 章节中文化

现有英文标题将全部改成中文表达：

- `Overview` -> `项目概述`
- `Problem` -> `核心问题`
- `Constraints` -> `项目约束`
- `Approach` -> `方案路径`
- `Key Decisions` -> `关键判断`
- `Tech Stack` -> `关键能力 / 技术要点`
- `Result & Impact` -> `结果与价值`
- `Learnings` -> `项目收获`

中文化的目标不是直译，而是更贴合作品集阅读语境。

### 7.4 信息层级

详情页中不同模块的层级会重新拉开：

- 页头是总览层
- 指标卡 / 状态标签是辅助证据层
- 正文是叙事层
- 相关推荐和返回入口是收口层

“关键判断”和“结果与价值”会继续保留，但版式会更像作品卡和结论卡，而不是长段技术记录。

## 8. 组件层调整

### 8.1 `ProjectCard.astro`

这个组件会承担项目列表页的主要改版工作：

- 改成浅色编辑式卡片
- 中文化按钮和标签
- 强化 featured / ongoing 等状态表现
- 兼容是否带原型 / 仓库 / 外链的提示信息

### 8.2 `BaseCard.astro`

如果 `BaseCard` 会影响别的组件，则尽量少动；如果必须改，优先只调整基础壳层样式，避免牵连过大。

### 8.3 `CaseStudyLayout.astro`

这个布局只在以下情况调整：

- 阅读宽度需要更适配中文案例页
- 标题层级需要统一
- 全局留白与模块节奏需要修正

## 9. 技术落点

预计主要涉及这些文件：

- `src/pages/projects/index.astro`
- `src/components/ProjectCard.astro`
- `src/components/BaseCard.astro`
- `src/pages/projects/[slug].astro`
- `src/layouts/CaseStudyLayout.astro`
- 必要时调整 `src/pages.config.ts`

实现方式：

- 不修改 `content.config` schema
- 不批量重写已有 `.mdx` 内容字段
- 优先通过页面模板和组件样式完成改版
- 原型 / GitHub / 外链入口继续沿用当前正文中的链接块，并在详情页头部补充前置入口

## 10. 可访问性与响应式要求

### 可访问性

- 详情页入口按钮必须可键盘聚焦
- 列表页卡片跳转区保持足够点击面积
- 标签与文本对比度足够
- 中文标题层级保持语义正确

### 响应式

- 项目列表页在桌面端为多列卡片，在移动端自然堆叠
- 项目详情页头部在移动端改为单列堆叠
- 入口按钮在移动端需要改成全宽或接近全宽

## 11. 风险控制

主要风险：

- 改得太花，削弱项目内容本身
- 组件层改动过大，影响其他内容页
- 详情页头部增加入口后，页面显得拥挤

控制方式：

- 继续沿用首页的暖白、描边、标签、胶囊按钮语言
- 优先在 `projects` 路由内完成变化
- 入口按钮只保留最关键的 1 到 3 个，不堆满页头

## 12. 实施顺序

建议按这个顺序实施：

1. 列表页 Hero 与整体版式
2. `ProjectCard` 重做
3. 详情页头部案例封面区
4. 详情页章节中文化与正文节奏调整
5. 入口按钮、返回入口和收口模块统一
6. `astro check` 校验
